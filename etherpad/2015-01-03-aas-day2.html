Git
Reasons to use version control as an individual and when working collaboratively. Track changes, fix mistakes, go back, have only one "final" copy, remote backup. Github can also be a way to share your code with others.

git config --global user.name "Your Name"
git config --global user.email "you@some.domain"
git config --global color.ui "auto"
git config --global core.editor "your_editor"

git status - gives you information about the current state of your repository. Use this a lot!
git add - tell git to keep track of a file. Git will not track files until explicitly told to do so.(other version control systems may work differently as far as tracking goes.)
git add - also used to include a changed file in the next commit, have to do this every time.
git commit - make a snapshot of your files. only includes files added via git add.
    Include a message with the -m flag, e.g. git commit -m "this is why I made these changes"
    (Don't forget the quotes around the message.)
git diff - shows you differences between current repo state and most recent commit
    Only includes tracked files, not new files.
git checkout - pull an older version of a file into your working directory
    Example: git checkout HEAD~1 mars.txt
    (git checkout is used for lots of things, it can get a bit confusing. you'll see it again.)
git remote - manage remote repositories that you talk to
    Example: git remote add remote_name url_of_remote_repository
    See what remotes you already have for a repository: git remote -v
git push - send your repo over the internet to a remote repository
    Example: git push origin master
    In that example "origin" is the name of the remote repository and
    "master" is the name of the branch you want to send.
    (More on branches later.)
git pull - retrieve changes from a remote repository and incorporate them into your local repo
    Example: git pull origin master
We often say "pull down" and "push up", meaning (from remote) and (to remote).

Referring to versions: HEAD is current, HEAD~1 the one before that, etc. Note it's tilde, not minus. The big long IDs like 392d1503 ("commit hashes") are absolute references which can be used (eg) when different people combine their repositories, whereas HEAD, HEAD~1 etc are relative references to whatever repository you're working in.

  • This says if you are running your code from the command line (python simcluster.py ...) then run the piece of code under this line (so in our example run the main function)
